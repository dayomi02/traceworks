# [프론트 가이드] 프로젝트 상세 — 요구사항 트리 렌더링

이 문서는 `GET /projects/{project_id}` 응답의 `requirements` 필드 구조가 평탄 리스트 → 2-depth 트리로 바뀜에 따라, 화면을 다시 그릴 때 참고할 수 있도록 작성한 가이드입니다. 특정 프레임워크를 가정하지 않으며 데이터 구조와 화면 패턴만 설명합니다.

---

## 1. 응답 구조 요약

`requirements`는 **대분류(Large) 배열**이며, 각 항목이 **`children`(중분류 Mid 배열)** 을 가집니다. 2단계 트리 이상의 깊이는 없습니다.

```jsonc
{
  "requirements": [
    {
      "req_id": "REQ-001",
      "req_name": "사용자 인증",
      "req_description": "…",
      "req_type": "기능",
      "user_type": ["사용자"],
      "req_priority": "HIGH",
      "req_status": "APPROVED",
      "children": [
        { "req_id": "REQ-001-001", "req_name": "이메일 로그인", "...": "..." },
        { "req_id": "REQ-001-002", "req_name": "비밀번호 재설정", "...": "..." }
      ]
    },
    {
      "req_id": "REQ-002",
      "req_name": "데이터 시각화",
      "children": []
    }
  ]
}
```

판별 키:
- **대분류**: `req_id` = `REQ-001` (토큰 2개)
- **중분류**: `req_id` = `REQ-001-001` (토큰 3개, 앞 두 토큰이 부모 대분류 `req_id`)

응답은 이미 정렬되어 있음 — 클라이언트에서 추가 정렬 불필요.

---

## 2. 화면 레이아웃 원칙

요구사항 영역은 **2단계 인덴트된 트리 뷰**로 표현합니다. 표/카드/아코디언 어느 형태든 다음 시각적 요소가 들어가면 됩니다.

### 2-1. 대분류 행
- **굵은 글씨** 또는 약간 큰 폰트
- 왼쪽에 토글 아이콘(▶ / ▼) — children이 있을 때만 노출
- 배경색을 살짝 강조(예: 옅은 회색) — 일반 행과 구분
- 우측에 children 개수 뱃지 (예: "3개")

### 2-2. 중분류 행
- 대분류 대비 **들여쓰기 16~24px**
- 일반 텍스트 굵기
- 왼쪽 끝에 트리 연결선(└─) 또는 점(•) — 부모-자식 시각화
- 토글 없음 (잎 노드)

### 2-3. 빈 children
- `children: []`인 대분류는 토글 아이콘을 숨기거나 비활성화 처리
- "하위 요구사항 없음" 같은 placeholder 텍스트는 **표시하지 않음** (UI 잡음)

---

## 3. 기본 인터랙션

| 동작 | 기대 동작 |
|------|----------|
| 대분류 행 클릭 | children 펼침/접기 토글 |
| 대분류 토글 아이콘 클릭 | 동일 (행 클릭과 동일) |
| 초기 진입 | **모든 대분류 펼친 상태**가 기본 (사용자가 한눈에 전체 파악) |
| "모두 접기" / "모두 펼치기" 버튼 | 상단에 옵션으로 제공 (필수 아님) |
| 중분류 행 클릭 | 상세 다이얼로그/사이드패널 오픈 (선택 사항) |

---

## 4. 필드별 표시 권장

| 필드 | UI 표현 |
|------|---------|
| `req_id` | 좌측 고정 폭 — 모노스페이스 폰트 추천 (예: `REQ-001` / `REQ-001-001`) |
| `req_name` | 메인 라벨, 한 줄 truncate (말줄임 …) |
| `req_description` | 두 번째 줄 회색 텍스트, 또는 행 클릭 시 상세에서 |
| `req_type` | 작은 뱃지 — `기능` (파랑), `비기능` (보라) |
| `user_type` | 다중 칩 (예: `사용자` `관리자`) |
| `req_priority` | 우선순위 뱃지 색상: `CRITICAL`=빨강, `HIGH`=주황, `MEDIUM`=노랑, `LOW`=회색 |
| `req_status` | 상태 뱃지 — `APPROVED` 등 |

대분류와 중분류는 **동일 필드 셋**을 갖지만, **중분류에는 우선순위/상태 뱃지를 생략**해 시각적 무게를 줄여도 됩니다.

---

## 5. 빈 상태 / 에러 처리

- `requirements: []` (요구사항 없음) → "등록된 요구사항이 없습니다." 안내 메시지만 노출
- `404` 응답 → 404 페이지로 라우팅
- `503` 응답 → "온톨로지 서비스에 일시적 문제가 있습니다. 잠시 후 다시 시도해주세요."

## 7. 구현 시 자주 묻는 질문

**Q. `req_id`만으로 부모/자식을 파싱해도 되나요?**
A. 백엔드가 이미 `children` 배열로 구조화해서 내려주므로 **클라이언트는 파싱하지 않아도 됩니다**. `requirements[*].children`을 그대로 신뢰하세요.

**Q. 중분류의 중분류(3-depth)가 생길 수 있나요?**
A. 현재 데이터 모델은 2-depth로 고정입니다. `children` 안의 항목은 `children` 필드를 가지지 않습니다.

**Q. 응답에 `children`이 없는 항목이 있을 수 있나요?**
A. 백엔드가 `children: []`로 항상 채워서 반환하므로 `undefined` 체크는 불필요합니다. 그래도 방어 코드를 두려면 `(item.children ?? []).length`로 처리하세요.

**Q. 정렬이 이미 되어 있다면 클라이언트 정렬은 필요 없나요?**
A. 네. 백엔드가 `req_id` 숫자 토큰 기준으로 정렬해서 보냅니다. 사용자 정의 정렬(우선순위순 등)이 필요할 때만 클라이언트에서 처리하세요.

**Q. 필터/검색은 어떻게 적용하나요?**
A. UI 단에서 트리를 평탄화한 임시 배열로 검색하고, 매칭된 노드와 **그 조상**을 함께 표시하는 방식을 권장합니다. 매칭된 중분류만 있고 부모 대분류가 검색어와 무관해도 부모를 같이 보여줘야 트리 구조가 깨지지 않습니다.

## 8. 백엔드 API 참고

- API 명세: [api-guide-project-detail.md](api-guide-project-detail.md)
- 응답 스키마: `ProjectDetail.requirements: list[RequirementGroup]`
  - `RequirementGroup` = `RequirementDetail` + `children: list[RequirementDetail]`
- 호출 예: `GET /projects/PRJ-001`

## 9. AI 코드 에이전트에게 작업을 시킬 때 사용할 프롬프트 예시

> "프로젝트 상세 화면의 요구사항 섹션을 다시 만들어줘.
>
> 데이터는 `GET /projects/{project_id}` 응답의 `requirements` 배열을 사용한다. 각 항목은 대분류 요구사항이고 `children` 필드에 중분류 배열이 들어있다 (2-depth 고정, 백엔드가 정렬·구성 완료).
>
> 화면 요구사항:
> - 대분류 행은 굵게, 토글 아이콘과 children 개수 뱃지를 노출
> - 중분류 행은 16~24px 들여쓰기 + 트리 연결선
> - 초기 진입 시 모든 대분류 펼침
> - 대분류 클릭 = 펼침/접기 토글
> - 필드별 표시는 `docs/api-guide/project-detail-requirements-ui-guide.md` 5번 표 참고
> - 빈 children인 대분류는 토글 비활성, '하위 없음' placeholder는 표시하지 않음
> - 필터/검색 시 매칭된 노드의 조상까지 함께 노출
>
> 기존 평탄 리스트 기반 코드는 무시하고, `requirements[*].children`을 그대로 신뢰하는 방식으로 새로 작성해줘."
