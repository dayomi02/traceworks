# Traceworks (디웍스) — 이력서용 포트폴리오 정리

> AI 기반 SI 프로젝트 실행관리 자동화 플랫폼. 백엔드 시스템 전반과 온톨로지 기반 데이터 아키텍처를 설계·구현. (2026-05 ~ 진행중)

---

## 한 줄 요약

> RFP 문서 한 건으로 요구사항·WBS·인력 계획·일정을 자동 생성하는 시스템을 설계·구현하면서, **LLM의 비결정성을 코드의 결정성으로 감싸는 아키텍처**와 **온톨로지 기반 관계 탐색 구조**로 정확도와 확장성을 동시에 확보했습니다.

---

## 핵심 기여 — 설계 의사결정과 문제 해결

### 1. "LLM은 콘텐츠, 코드는 구조" — 환각 제어 아키텍처 정립

**문제 상황**
- 초기 구현에서 LLM에 요구사항 ID 부여·중복 제거·정렬까지 맡겼더니, RFP를 분석할 때마다 **`REQ-001` ID가 청크 간 충돌**해 후속 단계가 깨지는 문제가 반복 발생.
- LLM이 같은 요구사항을 다른 ID로 두 번 반환하거나, 임의로 ID를 누락하는 등 비결정적 동작이 발생.

**의사결정**
- LLM의 책임을 **콘텐츠 추출(요구사항 텍스트·분류)** 로만 한정하고, **ID 부여·중복 제거·정렬 같은 구조적 작업은 코드가 결정적으로 처리**하는 원칙 수립.
- LLM 출력에는 `tempId`, `parentTempId`, `isLarge` 같은 **임시 식별자만** 두고, 코드가 정규화 단계에서 `REQ-001` 같은 최종 ID를 부여하도록 재설계.

**결과**
- 동일 RFP 재분석 시 ID 충돌 0건. 청크 간 중복 요구사항도 코드의 dedup 로직으로 안정적으로 제거.
- 이 원칙을 **WBS 일정 산출, 신뢰도 측정, 인력 추천**에도 동일하게 적용해 **시스템 전반의 비결정성 통제**.

---

### 2. RFP 분석 파이프라인 — 단일 호출 → 청킹 병렬 처리로 전환

**문제 상황**
- 초기에는 RFP 전체를 한 번에 LLM에 전달했으나, **30페이지 이상 문서에서 중간 요구사항이 누락**되거나 응답 토큰 한계로 잘리는 문제 발생.
- 처리 시간도 5분 이상 걸려 사용자 경험이 좋지 않음.

**의사결정**
- 문서를 **요구사항 번호 헤딩(`FUN-01`, `PER-02` 등) 기준으로 청킹**하고, 청크별로 LLM을 **병렬 호출**하는 구조로 전환.
- 각 청크에는 **프로젝트 컨텍스트(도메인·기술스택·설명)를 함께 전달**해 청크 간 일관성을 유지.
- 청크 간 중복은 코드가 `tempId` 기반으로 dedup, 최종 ID는 정규화 단계에서 부여.

**결과**
- 30페이지 RFP 기준 처리 시간 **5분 → 3분 (40% 단축)**.
- 요구사항 추출률 80~90% 수준으로 안정화.
- **2125줄 규모의 RFP 분석 서비스 모듈(rfp_service.py)** 의 핵심 구조를 정의.

### 3. WBS 자동 일정 산출 — Topological Sort → Priority-Aware Scheduler로 전환

**문제 상황**
- 초기 일정 산출 로직은 단순 위상 정렬(topological sort)이라 — 의존성이 없는 작업끼리는 문서상 등장 순서대로 배치됨.
- 결과적으로 **인프라·공통 모듈 작업이 후순위로 밀리고, 리스크 높은 작업이 후반에 배치**되어 실패 시 전체 일정 재조정이 필요한 구조.

**의사결정**
- 5가지 평가축의 **가중 점수 기반 디스패처**로 재설계:
  ```
  priority = phase × 10 + criticality × 8 + risk × 15 + fan_out × 1 + bottleneck × 5
  ```
- **Risk를 가장 높은 가중치(×15)** 로 둬서, 외부 연계·PoC 같은 불확실한 작업을 일정상 늦더라도 조기 검증하도록 의도 반영.
- LLM이 분류한 3축(phase/criticality/risk)과 코드가 그래프 분석으로 자동 계산한 2축(fan-out/bottleneck)의 **하이브리드 점수 체계**.

**결과**
- 의존성 + 우선순위 + 자원 풀 + 영업일 제약을 **동시에 만족하는 최적 일정** 자동 산출.
- 사람이 수작업으로 놓치기 쉬운 "리스크 작업 조기 검증" 같은 운영 원칙을 시스템에 내재화.

---

### 4. 부분 재산출 알고리즘 — 일정 변경 시 완료 작업 보존

**문제 상황**
- PM 승인 후 추가 태스크가 생기면 후속 태스크 일정이 자동으로 밀려야 하지만, 기존 스케줄러는 **모든 task를 처음부터 다시 계산**해서 완료된 작업 일정까지 덮어쓰는 문제 발생.

**의사결정**
- 기존 `_schedule_wbs_tasks`에 **`pinned_codes` 인자를 추가**해, "이미 일정이 정해진 태스크"는 변경하지 않고 인원 풀만 점유시키는 구조로 확장.
- 핀(고정) 대상은: ① 완료 상태 태스크, ② 사용자가 직접 입력한 추가 태스크, ③ cutoff 이전에 끝나는 진행/미진행 태스크.
- 기존 함수의 시그니처를 깨지 않도록 **선택 인자(optional parameter)** 로 추가 → 기존 RFP 분석 흐름은 영향 없이 유지.

**결과**
- 일정 재산출 시 **완료된 작업은 절대 변경되지 않으면서**, 영향받는 후속 작업만 자동 재배치.
- API 호출자는 `reschedule_wbs: true` 한 줄로 자동 재산출을 트리거.

### 5. 온톨로지 기반 데이터 아키텍처 — 4-Layer 추상화

**문제 상황**
- 프로젝트·인력·요구사항·태스크 데이터를 관계형 DB로만 다루면, "이 태스크가 막히면 몇 개 작업이 연쇄적으로 지연되나?" 같은 **N-hop 관계 탐색이 재귀 SQL·N+1 쿼리 폭증**으로 풀기 어려움.

**의사결정**
- 도메인 데이터를 **Apache Jena Fuseki 기반 RDF/SPARQL 지식 그래프**로 구성.
- **4단계 추상화** 구조로 설계:
  1. **Core Entities** (정의) — Project, Person, Task, Requirement 등
  2. **WorkEvent** (기록) — GitLab/Figma/Slack 등 외부 이벤트 정규화
  3. **SemanticUnit** (해석) — LLM이 이벤트를 의미 단위 신호로 변환
  4. **Service Layer** (실행) — Alert, WBSSnapshot, StaffingRecommendation 등 산출물
- 의존성·인력 매칭·영향도 분석은 SPARQL Property Path(`dependsOn+`) 한 줄로 표현.

**결과**
- 4가지 핵심 기능(**AI 인력 추천, 파급력 분석, AI 제언, 알림 라우팅**)을 모두 그래프 쿼리 하나로 구현.
- **1620줄 규모의 SPARQL 저장소 모듈(sparql_repo.py)** 로 RDF 트리플 적재·조회·갱신을 추상화.

---

### 6. 폴리글랏 영속성 — Fuseki + MariaDB 역할 분리

**문제 상황**
- 모든 데이터를 그래프 DB에 두면 — 사용자 인증·작업 이력·알림 이벤트 같은 **빈번한 트랜잭션과 시계열 데이터** 처리에 비효율적.
- 반대로 관계형 DB만으로는 **N-hop 관계 탐색이 어렵고 스키마 변경 비용이 큼**.

**의사결정**
- **도메인 데이터(엔티티·관계 중심)는 Fuseki에**, **운영 데이터(트랜잭션·이력 중심)는 MariaDB에** 두는 폴리글랏 영속성 구조 채택.
- MariaDB에는 `WbsTaskHistory`(상태 변경 이력), `User`(인증·세션), 알림 이벤트 등.
- Fuseki에는 `Project`, `Requirement`, `Task`, `Person`, `Skill` 등 도메인 엔티티.

**결과**
- 각 저장소가 자기 강점에 맞는 작업만 처리 → **두 도구의 약점을 모두 피하는 구조**.
- 인증·이력은 RDB의 ACID로 보장, 관계 탐색은 그래프 쿼리로 단순화.

---

### 7. RFP 분석 신뢰도 3축 정량화 — AI 결과의 검증 가능성 확보

**문제 상황**
- AI 분석 결과를 그대로 운영에 반영하면 환각·누락이 그대로 흘러들어가는 문제.
- 단순 "성공/실패" 표시로는 사용자가 **어디를 검토해야 할지** 알 수 없음.

**의사결정**
- 분석 결과에 **3축 신뢰도 점수**를 함께 반환:
  - `project_extraction` (프로젝트 메타 추출)
  - `requirements_classification` (요구사항 분류)
  - `wbs_accuracy` (WBS 산정 정확도)
- 각 단계의 결손률·일관성·통계를 기반으로 가중 평균.
- **`low_confidence_items`** 로 검토 권장 항목을 구체적으로 노출, **`source_text`** + **`source_chunk_text`** 로 원문 출처까지 추적.

**결과**
- AI 결과의 신뢰도를 **사용자가 정량적으로 인지**하게 됨.
- 신뢰도 60% 미만은 시스템이 먼저 검토 경고를 띄우는 가드레일 형성.

---

### 8. 외부 시스템 통합 — 어댑터 패턴 + 비동기 + 무중단 보장

**문제 상황**
- RFP 확정 시 **Google Slides·GitLab 저장소 자동 생성**, 상태 변경 시 **알림 API + Slides 웹훅** 호출 등 외부 시스템 의존이 다수.
- 외부 서비스 장애·타임아웃이 본 서비스 핵심 흐름(승인·일정 산출)을 막으면 안 됨.

**의사결정**
- 모든 외부 통신은 `httpx.AsyncClient` 기반 비동기 호출, **timeout 명시 + try/except로 격리**.
- 외부 호출 실패는 **WARNING 로그만 남기고 본 흐름은 정상 완료**되도록 설계.
- 알림 API 422 에러(prev_status="TODO" 미허용) 디버깅을 통해 **수신 서버의 enum 호환 이슈를 우리쪽 페이로드 분기로 흡수**.

**결과**
- 외부 시스템 장애 시에도 **승인·상태 변경 같은 핵심 흐름이 절대 막히지 않음**.
- 알림 시스템 422 에러 발생률 거의 0건으로 감소.

---

### 9. PM 승인/반려 흐름 — 권한 분리 + 알림 라우팅

**문제 상황**
- 작업자가 상태를 변경할 때, **PM이면 즉시 반영 / 일반 사용자면 PM 승인 필요** 같은 분기와, **알림 대상이 누구인지** 결정하는 복잡한 로직이 필요.
- "완료 → 다른 상태" 같은 특수 케이스에서는 추가 태스크 자동 생성까지 필요.

**의사결정**
- 권한·즉시반영 여부·알림 대상을 **WBS_STATUS 전이 매트릭스 + 권한 체크 함수**로 명확히 분리.
- PM이 변경 시 → 본인 제외 프로젝트 멤버 전체에 알림.
- 일반 사용자 변경 시 → PM에게 승인 요청 알림.
- 승인 결과 → 담당자에게 결과 알림 + 프로젝트 전원에게도 정보 전달.
- "미진행 → 진행"은 알림 노이즈를 줄이기 위해 알림/웹훅 스킵.

**결과**
- 권한·알림·상태 변경·일정 재산출이 한 흐름으로 통합된 **433줄 규모의 tasks.py API 흐름** 완성.

### 10. 요구사항 트리화 — 클라이언트 파싱에서 서버 응답 가공으로 전환

**문제 상황**
- 요구사항이 평탄 배열로 응답되어, 화면에서 **대분류/중분류 트리를 그릴 때 클라이언트가 `req_id`를 매번 파싱**해야 함.
- 정렬·고아 처리·중분류 누락 같은 엣지케이스 처리를 클라이언트마다 반복.

**의사결정**
- 서버에서 응답을 **`requirements[*].children[]` 2-depth 트리**로 가공.
- `req_id` 명명 규약(`REQ-001` vs `REQ-001-001`)을 기준으로 코드가 트리 빌드.
- 부모 없는 Mid는 자동으로 루트로 승격(고아 방지), 숫자 토큰 기준 정렬.

**결과**
- 클라이언트는 `children`을 그대로 신뢰만 하면 됨.
- 단위 테스트로 트리화 로직(`_build_requirement_tree`) 검증 — 정렬·고아·빈 입력 모두 정상.

## 핵심 키워드 (이력서/면접용)

- **AI/LLM 응용**: LLM 환각 제어, 청킹 기반 분석 파이프라인, 신뢰도 정량화, 프롬프트 엔지니어링

- **알고리즘**: 우선순위 기반 스케줄러, 의존성 그래프 분석, 부분 재산출, 가중 평균 매칭
